## Excel 파일 (`SAAS_PRICING_JP.xlsx`) 사용법

5개 시트로 구성됩니다. 黄色 셀 (¥/숫자/플랜명)에 값을 입력하면 緑色 셀이 자동 갱신됩니다.

### 시트 1. トップ (표지)

회사 정보 채워넣기:
- 提案先 → 고객 회사명
- 提案者 → 본인 회사명
- 作成日 → 제안 일자

### 시트 2. 見積シミュレーション (가격 시뮬레이터) ⭐

**가장 자주 쓸 시트.** 황색 셀에 입력만 하면:
- 店舗数 (예: 10)
- 選択プラン (Starter / Business / Enterprise)
- 契約年数 (1~5년)
- 年間契約割引 (10% 등)

→ 자동 계산:
- 月額合計
- 年額 (할인 적용 후)
- 契約年数 총액 (초기 비용 포함)

미팅 자리에서 화면 띄워놓고 **「점포 50개면 얼마인가요?」** → 즉시 숫자 갱신.

### 시트 3. プラン比較 (3 티어 비교표)

35행 비교표. 그대로 인쇄해서 미팅 자료로 사용. 첫 행 + 첫 열은 자동 고정 (스크롤해도 보임).

### 시트 4. 5年TCO比較 (자체 개발 vs SaaS) ⭐⭐

**가장 강력한 영업 카드.**

전제 조건 (점포 수, 운용 연수, 자체 개발 견적 추정 등) 황색 셀에서 조정 가능. 결과:
- 自社開発 (中堅 IT) 5년: ¥85,000,000
- 自社開発 (大手 SIer) 5년: ¥225,000,000
- **WorkBridge SaaS 5년: ¥27,200,000**
- 비율: **1/3 ~ 1/8 가격**

이 숫자를 보여주면 「¥9,800/점포가 비싸다」 라는 반론이 즉시 봉쇄됩니다.

### 시트 5. オプション (애드온 가격표)

12개 옵션 (언어 추가, POS 연동, 프리미엄 SLA 등) 명세. 별도 견적 시 참조용.

## 가격 변경이 필요할 때 (¥9,800 → ¥10,800 같은)

두 가지 방법:

**방법 A — Excel 파일 직접 편집**
- `SAAS_PRICING_JP.xlsx`를 열어 황색 입력 셀 값 수정
- 다른 셀은 수식이라 자동 갱신
- 「ファイル → 名前を付けて保存」으로 별도 버전 보관

**방법 B — `_build_xlsx.py` 스크립트로 재생성**
- 환율·API 단가 같은 「전제값」 변경 시 권장
- 스크립트 안의 숫자 (예: `9800`, `12800`) 수정 후:
  ```
  python proposal/_build_xlsx.py
  ```
- 깨끗한 새 파일 생성. Git으로 변경 이력 관리

## Google 문서로 변환하는 방법 (가장 권장)

### 방법 A — 직접 복사·붙여넣기 (가장 빠름)

1. `SAAS_PROPOSAL_JP.md`를 텍스트 에디터 (VS Code, Notepad++, 메모장 등)로 엽니다
2. **Ctrl+A** → **Ctrl+C** 로 전체 복사
3. Google Docs (https://docs.google.com)에 새 문서 만들기
4. **Ctrl+V** 로 붙여넣기 → 표·헤딩이 자동 변환됨
5. 아래 「채워넣기 항목」 의 `[]` 부분을 실제 정보로 교체

### 방법 B — Markdown → DOCX 변환 후 업로드

Pandoc 등 도구를 쓸 수 있다면:
```
pandoc SAAS_PROPOSAL_JP.md -o proposal.docx
```
생성된 `.docx`를 Google Drive에 업로드하면 자동 변환됩니다.

### 방법 C — GitHub Markdown 미리보기에서 복사

이 파일을 GitHub에 푸시한 뒤, GitHub의 렌더링된 화면에서 그대로 복사하면 표가 깔끔하게 보존되며 Google Docs로 옮겨집니다.

## 사용 시나리오

### A. PoPo (대리점) 에 보낼 때

1. 「提案先」 → 「[PoPo様向け 営業ツール]」 같이 표기
2. 「11. 次のステップ」 부분을 PoPo가 클라이언트에 다시 전달할 수 있게 약간 일반화 (이미 그렇게 되어 있음)
3. 가격은 **소매가** (PoPo가 클라이언트에 청구할 가격) 그대로 유지. PoPo의 수수료 분배는 별도 SOW에 명시
4. PDF로 export 후 메일 첨부

### B. C社 (직접 고객) 에 보낼 때

1. 표지에 C社 정보 채우기
2. 6 절 「導入規模別シミュレーション」에서 C社 규모에 맞는 시나리오 (A/B/C/D) **하나만** 남기고 나머지 삭제하면 더 명확
3. C社가 「내부 사용만」이라면 「再販不可」 임을 11절 등에 추가 명시 가능
4. 별첨: SOW (별도 작성), DPA (필요 시), SECURITY.md

### C. 새 잠재 고객 일반 영업

기본형 그대로 사용. 4 시나리오 모두 보여주는 것이 「우리는 다양한 규모를 다룬다」 라는 메시지가 됨.

### 계약 단계 송부 패키지 예시 (LinguaSync C社)

```
0. NDA_LINGUASYNC_JP.md         ← 가장 먼저 (정보 공유 전)  ⭐ 추가
1. PROPOSAL_LINGUASYNC_JP.md   ← 영업·검토용
2. PRICING_LINGUASYNC_JP.xlsx  ← 시뮬레이터로 합의 도달
3. SECURITY.md                  ← 보안팀 검토용
4. SOW_LINGUASYNC_JP.md         ← 계약 본문 (서명용)
5. DPA_LINGUASYNC_JP.md         ← 개인정보 처리 위탁 (서명용)
```

### NDA 체결 타이밍

NDA 는 「상세한 견적·고객 정보·데모」 가 오가기 **전** 에 체결하는 것이 원칙. 일본 대기업·상장사 영업에서는 거의 필수입니다. 「데모 전에 NDA 부터 보내주세요」 라고 먼저 제안하면 「제대로 된 회사」 라는 인상이 남습니다.

### 4 + 5 (SOW + DPA) 의 관계

| 문서 | 다루는 내용 |
|---|---|
| SOW | 작업 범위·일정·비용·납품물·SLA·해약 등 비즈니스 조건 |
| DPA | 개인정보의 취급·재위탁·국외이전·누출시 통지 등 데이터 보호 |

→ 두 문서는 같은 날 서명하는 것이 일반적. SOW §12 (個人情報) 과 DPA 본문이 모순되지 않도록 정합성 점검 필수.
